KEY RELATIONSHIPS
================================================================================

1. INHERITANCE:
   - Student extends User
   - CareerCenterStaff extends User
   - CompanyRepresentative extends User

2. COMPOSITION:
   - Internship contains List<InternshipApplication>
   - Student contains List<InternshipApplication>
   - CompanyRepresentative contains List<Internship>
   - DataManager contains Maps of all entities

3. ASSOCIATION:
   - InternshipApplication references Student and Internship
   - Internship references CompanyRepresentative (as repInCharge)
   - StudentManager, StaffManager, CompanyRepManager all use DataManager
   - All Managers use AuthenticationManager for user management

4. DEPENDENCY:
   - All View classes depend on Control layer managers
   - All Managers depend on DataManager for data persistence
   - AuthenticationManager depends on DataManager for user lookup

5. MULTIPLICITY:
   - 1 Student : 0..3 InternshipApplications
   - 1 CompanyRepresentative : 0..5 Internships
   - 1 Internship : 0..n InternshipApplications
   - 1 Internship : 1 CompanyRepresentative (repInCharge)


DESIGN PATTERNS USED
================================================================================

1. MVC (Model-View-Controller):
   - Model: Entity classes (User, Student, Internship, etc.)
   - View: View classes (LoginFrame, StudentDashboard, etc.)
   - Controller: Manager classes (StudentManager, StaffManager, etc.)

2. Singleton Pattern (implied):
   - DataManager acts as a central data repository

3. Factory Pattern:
   - InternshipLevel uses factory method (fromShortCode)

4. Strategy Pattern:
   - Multiple Manager classes for different user types

5. Observer Pattern (implicit):
   - DataManager notifies changes to all managers
